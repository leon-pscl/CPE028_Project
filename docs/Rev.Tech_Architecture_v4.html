1. System Overview

Rev.Tech guides Philippine consumers through a structured repair-or-recycle decision for defective or end-of-life smartphones and laptops, then connects them to verified local resources. Three cooperating subsystems — Assess, Navigate, and Connect — cover the full user journey. A one-time seeding pipeline initialises the directory; ongoing self-registration with admin verification keeps it current.

4. Assess Subsystem IPO

flowchart TD A1["Brand + model"] --> P1 A2["Age in months"] --> P1 A3["Issue category + severity"] --> P1 A4["Repair history flag"] --> P1 P1["① Lookup device specs (devices table)"] --> P2 P2["② Weighted factor matrix\nage · issue · parts avail · repairability · env-impact"] --> P3 P3["③ Normalise 0–100\nrepair ≥50 · recycle <50"] --> P4 P4["④ Cost-estimate heuristic\nmedian parts + labour per model"] --> P5 P5["⑤ Confidence scorer\ndata completeness ratio"] --> P6 P6["⑥ Rationale template engine\nplain-language string builder"] --> P7 P7["⑦ Persist: assessments + repair_scores + cost_estimates"] P7 --> O1["Score 0–100 + direction\nREPAIR / RECYCLE"] P7 --> O2["Cost range (PHP)"] P7 --> O3["Confidence HIGH/MED/LOW"] P7 --> O4["Plain-language rationale ≤3 sentences"] P7 --> O5["assessment_id → Navigate"]

Assess IPO. The weighted factor matrix is the core business rule; the rationale template engine produces consumer-readable output without exposing raw factor scores.

Inputs
  • Device brand (string, required)
  • Device model (string, required)
  • Age in months (integer)
  • Issue: battery / screen / motherboard / liquid / other
  • Severity: minor / moderate / severe
  • Prior repair attempts (boolean)
  • User auth token (optional)
Processes
  • Device lookup → device_id, category, repairability_idx
  • Factor scoring: age 20% · severity 30% · cost_ratio 25% · parts_avail 15% · mfr_support 10%
  • Score normalisation 0–100
  • Cost heuristic: median(parts) + median(labour) per model+issue
  • Confidence: DB record completeness → HIGH / MEDIUM / LOW
  • Template interpolation by score band
  • DB write assessments + scores + estimates
Outputs
  • assessment_id (UUID)
  • score: integer 0–100
  • direction: REPAIR | RECYCLE
  • cost_min, cost_max (PHP)
  • confidence: HIGH | MEDIUM | LOW
  • rationale: string ≤3 sentences
  • created_at timestamp
Scoring note (Phase 1 rule-based baseline): Defaults: age 20%, issue severity 30%, cost-vs-value ratio 25%, parts availability 15%, manufacturer support 10%. Stored in scoring_config table — tunable via admin UI without code deploys. Serves as ML fallback and academic benchmark. From Phase 1.5 onward, scores are produced by the ML service (see §16–19); the rule-based formula activates automatically if the ML service is unavailable.

5. Navigate Subsystem IPO

flowchart TD N1["assessment_id + direction"] --> NP1 N2["User auth token"] --> NP1 NP1["① Fetch roadmap template\n(repair or recycle path)"] --> NP2 NP2["② Hydrate steps with device content\n(DIY guide URLs, data-wipe guide)"] --> NP3 NP3["③ Load persisted checklist state\nfrom Supabase checklist_completions"] --> NP4 NP4["④ Stateful checklist\nTODO → DONE | SKIPPED"] --> NP6 NP6["⑤ Sync step completion → Supabase checklist_completions\n(idempotent write)"] --> NO1 NP4 --> NP8["⑥ Referral trigger\n→ Connect subsystem"] NP8 --> NO2["Connect: pre-filtered\nby device + direction"] NP1 --> NO3["DIY guide signed URL\n(Supabase Storage, 1h TTL)"] NO1["Roadmap UI with persisted step state"]

Navigate IPO. Checklist state is persisted directly to Supabase on each step completion (idempotent). NP8 is the primary handoff to Connect. No offline queue or Background Sync required — browser connectivity assumed.

Checklist persistence: Step completions are written to checklist_completions via an idempotent POST on each tap. On reload, state is restored from Supabase. No local storage or offline queue needed.

Connect IPO. Both ingestion paths (self-registration and scraping) funnel through the verification workflow. Automated checks reduce human review burden.

Compliance & ethical note (scraping): Scraping targets publicly listed business information only. Raw data stored in isolated staging store — never transmitted to users directly. Admin must approve each record before live directory insertion. Collect only: business name, address, phone, service categories. 10. Database Schema

All tables in public schema unless noted. PostGIS geometry(Point,4326) type assumed available.

-- users (extended from Supabase Auth)
CREATE TABLE users (
  id         UUID PRIMARY KEY DEFAULT auth.uid(),
  email      TEXT UNIQUE NOT NULL,
  full_name  TEXT,
  role       TEXT NOT NULL DEFAULT 'consumer'
             CHECK (role IN ('consumer','shop_admin','verifier','super_admin')),
  created_at TIMESTAMPTZ DEFAULT now()
);

-- devices
CREATE TABLE devices (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  brand             TEXT NOT NULL,
  model             TEXT NOT NULL,
  category          TEXT NOT NULL CHECK (category IN ('smartphone','laptop')),
  release_year      INTEGER,
  repairability_idx NUMERIC(3,2),          -- 0.00–1.00
  parts_availability TEXT CHECK (parts_availability IN ('HIGH','MEDIUM','LOW','EOL')),
  UNIQUE (brand, model)
);
CREATE INDEX idx_devices_brand_model ON devices (brand, model);

-- assessments
CREATE TABLE assessments (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id        UUID REFERENCES users(id) ON DELETE SET NULL,
  device_id      UUID REFERENCES devices(id),
  age_months     INTEGER NOT NULL,
  issue_category TEXT NOT NULL,
  issue_severity TEXT NOT NULL CHECK (issue_severity IN ('minor','moderate','severe')),
  prior_repairs  BOOLEAN DEFAULT FALSE,
  created_at     TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_assessments_user ON assessments (user_id);

-- repair_scores
CREATE TABLE repair_scores (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  assessment_id  UUID REFERENCES assessments(id) ON DELETE CASCADE,
  score          INTEGER NOT NULL CHECK (score BETWEEN 0 AND 100),
  direction      TEXT NOT NULL CHECK (direction IN ('REPAIR','RECYCLE')),
  confidence     TEXT NOT NULL CHECK (confidence IN ('HIGH','MEDIUM','LOW')),
  rationale      TEXT NOT NULL,
  config_version TEXT NOT NULL  -- references scoring_config.version
);

-- cost_estimates
CREATE TABLE cost_estimates (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  assessment_id UUID REFERENCES assessments(id) ON DELETE CASCADE,
  cost_min_php  INTEGER NOT NULL,
  cost_max_php  INTEGER NOT NULL,
  currency      TEXT DEFAULT 'PHP',
  basis         TEXT    -- e.g. 'parts_db_median + labour_regional'
);

-- guides
CREATE TABLE guides (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title        TEXT NOT NULL,
  device_id    UUID REFERENCES devices(id),
  guide_type   TEXT CHECK (guide_type IN ('DIY_REPAIR','DATA_WIPE','RECYCLE_PREP')),
  storage_path TEXT NOT NULL,   -- Supabase Storage path
  version      TEXT NOT NULL DEFAULT '1.0',
  published_at TIMESTAMPTZ
);

-- shops (repair shops)
CREATE TABLE shops (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name            TEXT NOT NULL,
  address         TEXT NOT NULL,
  city            TEXT NOT NULL,
  province        TEXT NOT NULL,
  geom            GEOMETRY(Point, 4326),    -- PostGIS
  phone           TEXT,
  website         TEXT,
  hours           JSONB,                    -- {"mon":"8:00-18:00",...}
  device_types    TEXT[] DEFAULT '{}',
  brands_serviced TEXT[] DEFAULT '{}',
  verified        BOOLEAN DEFAULT FALSE,
  verified_at     TIMESTAMPTZ,
  registered_by   UUID REFERENCES users(id),
  source          TEXT DEFAULT 'self_registration'
                  CHECK (source IN ('self_registration','scraping','manual_admin')),
  created_at      TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_shops_geom ON shops USING GIST(geom);       -- PostGIS
CREATE INDEX idx_shops_verified ON shops (verified);

-- facilities (recycling / TSD)
CREATE TABLE facilities (
  id                   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name                 TEXT NOT NULL,
  address              TEXT NOT NULL,
  city                 TEXT, province TEXT,
  geom                 GEOMETRY(Point, 4326),
  facility_type        TEXT CHECK (facility_type IN ('recycling_center','TSD','brand_takeback','drop_off')),
  denr_registration_no TEXT,               -- DENR TSD registration number
  ra11898_program      TEXT,               -- brand name for take-back programs
  accepts_devices      TEXT[] DEFAULT '{}',
  verified             BOOLEAN DEFAULT FALSE,
  verified_at          TIMESTAMPTZ,
  cert_doc_path        TEXT,               -- Supabase Storage path
  source               TEXT DEFAULT 'self_registration',
  created_at           TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_facilities_geom ON facilities USING GIST(geom);
CREATE INDEX idx_facilities_denr ON facilities (denr_registration_no)
  WHERE denr_registration_no IS NOT NULL;

-- certifications
CREATE TABLE certifications (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  entity_id   UUID NOT NULL,
  entity_type TEXT NOT NULL CHECK (entity_type IN ('shop','facility')),
  cert_type   TEXT NOT NULL,     -- 'DENR_TSD','RA11898','ISO14001'
  cert_number TEXT,
  issued_by   TEXT,
  valid_from  DATE, valid_until DATE,
  doc_path    TEXT,              -- Supabase Storage
  created_at  TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_certs_entity ON certifications (entity_id, entity_type);

-- verification_tasks
CREATE TABLE verification_tasks (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  entity_id        UUID NOT NULL,
  entity_type      TEXT NOT NULL CHECK (entity_type IN ('shop','facility')),
  status           TEXT NOT NULL DEFAULT 'PENDING'
                   CHECK (status IN ('PENDING','AUTO_PASS','AUTO_FAIL','UNDER_REVIEW','APPROVED','REJECTED')),
  auto_checks      JSONB DEFAULT '{}',  -- {"geocode_ok":true,"denr_match":false}
  assigned_to      UUID REFERENCES users(id),
  reviewed_at      TIMESTAMPTZ,
  rejection_reason TEXT,
  created_at       TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_vtasks_status ON verification_tasks (status);

-- audit_logs
CREATE TABLE audit_logs (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  actor_id    UUID REFERENCES users(id),
  action      TEXT NOT NULL,
  entity_id   UUID, entity_type TEXT,
  metadata    JSONB DEFAULT '{}',
  ip_address  INET,
  created_at  TIMESTAMPTZ DEFAULT now()
);
-- Retention: 2 years; partition by month at scale
CREATE INDEX idx_audit_entity ON audit_logs (entity_id, entity_type);

-- staging.seeding_sources (isolated schema)
CREATE TABLE staging.seeding_sources (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  source_type TEXT NOT NULL CHECK (source_type IN ('google_maps','facebook_page','manual')),
  source_url  TEXT,
  raw_data    JSONB NOT NULL,
  scraped_at  TIMESTAMPTZ NOT NULL,
  run_id      TEXT,             -- GitHub Actions run ID
  imported    BOOLEAN DEFAULT FALSE,
  imported_at TIMESTAMPTZ,
  imported_by UUID              -- admin user ID
);
-- Retention: 90 days unimported; 1 year for imported records

-- checklist_completions
CREATE TABLE checklist_completions (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  assessment_id    UUID REFERENCES assessments(id) ON DELETE CASCADE,
  step_id          TEXT NOT NULL,
  status           TEXT NOT NULL DEFAULT 'DONE'
                   CHECK (status IN ('DONE','SKIPPED')),
  idempotency_key  TEXT UNIQUE NOT NULL,  -- prevents duplicate writes
  completed_at     TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_checklist_assessment ON checklist_completions (assessment_id);

-- user_transactions (aggregate ledger for user-facing history)
-- Materialises key events across assessments, roadmaps, and outcomes
-- so /profile/history can be served from a single query.
CREATE TABLE user_transactions (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id        UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  assessment_id  UUID REFERENCES assessments(id) ON DELETE SET NULL,
  event_type     TEXT NOT NULL
                 CHECK (event_type IN ('ASSESSMENT_CREATED','ROADMAP_STARTED',
                        'ROADMAP_COMPLETED','OUTCOME_SUBMITTED','SHOP_REGISTERED')),
  payload        JSONB DEFAULT '{}',  -- snapshot: score, direction, device name, etc.
  created_at     TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX idx_user_tx_user ON user_transactions (user_id, created_at DESC);
-- Written inside a DB transaction alongside the primary record write to ensure
-- history is never out of sync. Use Supabase DB triggers or server-side
-- BEGIN/COMMIT blocks in the Scoring Fn and Navigate Fn.


  version    TEXT PRIMARY KEY,
  weights    JSONB NOT NULL,   -- {"age":0.25,"issue_severity":0.30,...}
  active     BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMPTZ DEFAULT now()
);
Transaction integrity: Writes that span multiple tables (e.g. assessments + repair_scores + cost_estimates + user_transactions) must be wrapped in a single BEGIN / COMMIT block executed from the server-side Scoring Fn using the Supabase service-role connection. This prevents partial writes where a score record exists without its parent assessment. Use supabase.rpc('create_assessment_tx', { ... }) calling a Postgres function, or issue raw SQL through the service-role client with explicit transaction control. Never write these tables in separate sequential API calls from the client.
User-facing history (user_transactions): The user_transactions table is a write-optimised event ledger — one row per significant user action. It is written inside the same DB transaction as the primary record so history is never out of sync. The payload JSONB column stores a denormalised snapshot (device name, score, direction) so the /profile/history page can be served from a single indexed query on (user_id, created_at DESC) without JOINs across multiple tables.

Prefixed /api/v1. Supabase REST also available at /rest/v1/. Auth via Supabase JWT Bearer.

11.0 Authentication

Auth is handled by Supabase Auth directly from the client — the routes below are the Supabase Auth SDK calls, not custom Vercel functions. They are documented here for completeness and for the agent's reference.

POST/auth/v1/signupauth: none
Register with email + password. Supabase sends a confirmation email. On confirmation, a DB trigger fires inserting the user into public.users with role consumer.
// Request (Supabase client SDK: supabase.auth.signUp)
{ "email": "user@example.com", "password": "••••••••",

16. ML-Driven Scoring — Approach & Decision Logic

The Repair-or-Recycle Score is produced by a machine learning model rather than a static formula. The ML approach is chosen based on dataset availability at the time of training, with a defined upgrade path as real-world data accumulates.

Path Selection

ConditionApproachAlgorithmWhen
Labeled dataset available (≥ 500 confirmed repair/recycle outcomes)Binary ClassificationLogistic Regression → Random Forest → XGBoostPhase 2 onward
No labeled datasetClusteringK-Means or DBSCANPhase 1 (launch)
TransitionCluster labels → supervised classifierLogistic Regression on cluster-labeled dataPhase 1.5

Phase 1 — Clustering (No Labeled Dataset)

At launch, no ground-truth repair/recycle outcome data exists for Philippine devices. K-Means or DBSCAN is applied to device feature vectors to discover natural groupings, which are then interpreted and labeled by domain experts (repair technicians).

flowchart TD F1["device_age_months"] --> FV F2["issue_severity (encoded 1–3)"] --> FV F3["parts_availability (encoded 0–3)"] --> FV F4["repairability_idx (0.00–1.00)"] --> FV F5["estimated_cost_ratio (repair / value)"] --> FV F6["manufacturer_support (boolean)"] --> FV FV["Feature Vector"] --> SCALE["StandardScaler normalisation"] SCALE --> ALGO{"Algorithm"} ALGO -->|"k unknown"| DBSCAN["DBSCAN\n(density-based, handles outliers)"] ALGO -->|"k estimated via elbow"| KMEANS["K-Means\n(centroid-based)"] DBSCAN --> EVAL["Silhouette Score validation"] KMEANS --> EVAL EVAL --> LABEL["Domain expert labels clusters\n(REPAIR / RECYCLE / BORDERLINE)"] LABEL --> OUT["direction + confidence output"]

DBSCAN is preferred when k is unknown — it treats outlier devices as noise (BORDERLINE) rather than forcing them into a cluster. Silhouette score validates cluster quality before expert labeling.

Expert validation requirement: Before deploying cluster-based scoring, at least one certified repair technician must review cluster profiles and confirm that cluster-to-label mappings are technically sound. This must be documented in the model registry (ml_models.notes) and included in the academic methodology section.

Phase 2 — Binary Classification (Labeled Dataset)

Once user_confirmed_outcome data accumulates from post-assessment follow-ups, a supervised binary classifier replaces the clustering model. The classifier predicts REPAIR (1) or RECYCLE (0) from the same feature vector.

flowchart LR subgraph TRAIN["Training Pipeline (offline, Python)"] D1["assessments + repair_scores\n(historical)"] --> JOIN D2["user_confirmed_outcome\n(ground truth labels)"] --> JOIN JOIN["JOIN on assessment_id"] --> FEAT["Feature engineering\n+ StandardScaler"] FEAT --> SPLIT["Train / Validation / Test split\n(70 / 15 / 15)"] SPLIT --> LR["Logistic Regression\n(under 500 rows)"] SPLIT --> RF["Random Forest\n(500–5,000 rows)"] SPLIT --> XGB["XGBoost\n(5,000+ rows)"] LR & RF & XGB --> EVAL2["Evaluation\nAccuracy · F1 · AUC-ROC\nConfusion matrix"] EVAL2 --> EXPORT["Export best model\n(ONNX or pickle)"] EXPORT --> REGISTRY["INSERT ml_models\n(version, algorithm, accuracy, active=false)"] REGISTRY --> PROMOTE["Admin promotes to active=true\n→ FastAPI reloads model"] end

Model training runs offline in Python (scikit-learn / XGBoost). The exported model is versioned in ml_models and promoted to active by an admin — never auto-promoted. Logistic Regression is the first target: interpretable, auditable, and suitable for small datasets.

Label leakage warning: Do NOT use rule-based v1 outputs as training labels. Only user_confirmed_outcome (what the user actually did) qualifies as ground truth. Using rule outputs as labels would cause the classifier to simply learn the rules back, not real-world repair decisions.

17. ML Serving Architecture

The ML model is served by a dedicated FastAPI microservice, separate from the main Vercel application. This isolates the Python ML runtime, allows independent redeployment on model updates, and keeps the scoring latency budget predictable.

graph TB subgraph CLIENT["CLIENT"] UI2["React UI"] end subgraph VERCEL["VERCEL"] SCORE2["Scoring Fn\n(Vercel Serverless)"] end subgraph ML["ML SERVICE — Railway (Python)"] FAPI["FastAPI\n/predict endpoint"] MODEL["Loaded model\n(sklearn / ONNX)"] FAPI --> MODEL end subgraph DATA2["SUPABASE"] PG2["PostgreSQL\n(assessments, ml_models,\nfeature_vectors)"] SCFG["scoring_config\n(weights fallback)"] end UI2 -->|"POST /api/v1/assessments"| SCORE2 SCORE2 -->|"GET device record"| PG2 SCORE2 -->|"POST /predict {feature_vector}"| FAPI FAPI -->|"REPAIR/RECYCLE + confidence\n+ feature_importances"| SCORE2 SCORE2 -->|"INSERT assessment\n+ repair_score\n+ feature_vector snapshot"| PG2 SCORE2 -->|"score JSON"| UI2 SCORE2 -.->|"Fallback if ML service down"| SCFG

The Scoring Fn acts as orchestrator: it fetches device data, builds the feature vector, calls FastAPI /predict, then writes results to Supabase. If the ML service is unavailable, it falls back to the rule-based weighted formula (scoring_config table) — zero user-facing downtime.

FastAPI /predict Contract

// Request
{
  "device_age_months": 24,
  "issue_severity": 2,          // 1=minor 2=moderate 3=severe
  "parts_availability": 3,      // 0=EOL 1=LOW 2=MED 3=HIGH
  "repairability_idx": 0.72,    // 0.00–1.00 from devices table
  "cost_ratio": 0.18,           // repair_cost_estimate / device_value
  "manufacturer_support": 1     // 1=active 0=ended
}

// Response 200
{
  "direction": "REPAIR",
  "confidence": "HIGH",         // HIGH / MEDIUM / LOW (from predict_proba)
  "probability": 0.87,          // P(REPAIR) from classifier
  "feature_importances": {      // for rationale template engine
    "cost_ratio": 0.34,
    "issue_severity": 0.28,
    "parts_availability": 0.18,
    "repairability_idx": 0.12,
    "device_age_months": 0.06,
    "manufacturer_support": 0.02
  },
  "model_version": "v2.1-rf-2025-09"
}

The feature_importances field drives the rationale template engine — the top 2 factors are surfaced in the plain-language explanation shown to the user. For clustering (Phase 1), probability is replaced by a silhouette-derived confidence score: HIGH if silhouette ≥ 0.6, MEDIUM if 0.4–0.59, LOW if < 0.4 or device falls in a BORDERLINE cluster. The feature_importances field is omitted in clustering responses (replace with cluster centroid distances if interpretability is required for the academic evaluation). The Scoring Fn must handle both response shapes.

Hard override rules (applied in Scoring Fn, not ML model): Certain combinations force RECYCLE regardless of model output: (1) motherboard failure + age > 48 months, (2) severe liquid damage + age > 36 months, (3) parts_availability = EOL + issue_severity = severe. These are business rules, not ML decisions — applied post-prediction in the Scoring Fn before writing to DB.

-- ML model registry
CREATE TABLE ml_models (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  version      TEXT NOT NULL UNIQUE,   -- e.g. 'v1.0-kmeans', 'v2.1-rf-2025-09'
  algorithm    TEXT NOT NULL,          -- 'kmeans','dbscan','logistic_regression',
                                       --  'random_forest','xgboost'
  approach     TEXT NOT NULL CHECK (approach IN ('clustering','classification')),
  accuracy     NUMERIC(5,4),           -- validation accuracy (null for clustering)
  f1_score     NUMERIC(5,4),           -- weighted F1 (null for clustering)
  auc_roc      NUMERIC(5,4),           -- AUC-ROC (null for clustering)
  silhouette   NUMERIC(5,4),           -- silhouette score (null for classification)
  trained_at   TIMESTAMPTZ NOT NULL,
  active       BOOLEAN DEFAULT FALSE,
  artifact_url TEXT,                   -- Supabase Storage path to model file
  notes        TEXT                    -- expert validation notes (required for clustering)
);
-- Enforce single active model at all times
CREATE UNIQUE INDEX idx_ml_models_active ON ml_models (active) WHERE active = TRUE;

-- Feature vector snapshot (attached to each repair_score)
-- Critical for retraining: preserves exact inputs used at prediction time
ALTER TABLE repair_scores
  ADD COLUMN ml_model_id        UUID REFERENCES ml_models(id),
  ADD COLUMN feature_vector     JSONB,
  ADD COLUMN probability        NUMERIC(5,4),
  ADD COLUMN feature_importances JSONB,
  ADD COLUMN user_confirmed_outcome TEXT
    CHECK (user_confirmed_outcome IN ('REPAIRED','RECYCLED','REPLACED','NO_ACTION','PENDING'));

-- Ground truth collection: post-assessment follow-up survey
CREATE TABLE outcome_followups (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  assessment_id   UUID REFERENCES assessments(id) ON DELETE CASCADE,
  repair_score_id UUID REFERENCES repair_scores(id) ON DELETE CASCADE,
  outcome         TEXT NOT NULL
    CHECK (outcome IN ('REPAIRED','RECYCLED','REPLACED','NO_ACTION')),
  followed_up_at  TIMESTAMPTZ DEFAULT now(),
  channel         TEXT DEFAULT 'in_app'
    CHECK (channel IN ('in_app','email','push'))
);
CREATE INDEX idx_followups_assessment ON outcome_followups (assessment_id);

-- Impact tracking (SDG metric materialisation)
CREATE TABLE impact_events (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  assessment_id UUID REFERENCES assessments(id) ON DELETE SET NULL,
  event_type    TEXT NOT NULL CHECK (event_type IN ('REPAIR_COMPLETED','RECYCLE_COMPLETED')),
  device_id     UUID REFERENCES devices(id),
  device_weight_kg NUMERIC(5,3),   -- seeded from devices table avg weight
  recorded_at   TIMESTAMPTZ DEFAULT now()
);
-- Aggregate view for SDG dashboard
CREATE VIEW sdg_impact AS
  SELECT
    COUNT(*) FILTER (WHERE event_type = 'REPAIR_COMPLETED')   AS devices_repaired,
    COUNT(*) FILTER (WHERE event_type = 'RECYCLE_COMPLETED')  AS devices_recycled,
    ROUND(SUM(device_weight_kg), 2)                           AS ewaste_diverted_kg,
    DATE_TRUNC('month', recorded_at)                          AS month
  FROM impact_events
  GROUP BY DATE_TRUNC('month', recorded_at)
  ORDER BY month DESC;

21. Security Checklist

Auth & Authorisation

  • RLS enabled on ALL tables — never rely solely on application-level checks
  • Service-role key accessible only in server-side Vercel functions; absent from client bundle
  • MFA recommended (enforce) for verifier and super_admin roles via Supabase Auth TOTP
  • Disable implicit grant flow; access token 1 h; refresh token 7 days

Data Privacy (Philippine Data Privacy Act / GDPR-aligned)

  • PII minimisation: assessments store only device + issue data; no IMEI or device serial numbers
  • Anonymous assessments supported (user_id optional); user association is opt-in
  • Implement data erasure endpoint: DELETE /api/v1/users/me → nullify user_id FKs → delete user record
  • Cert documents in private Storage bucket; accessible only via 1-hour signed URLs
  • Data-wipe guide delivered as static PDF; no device data transmitted during this flow
  • Consent checkbox required before shop registration; stored in registrations table
  • Audit log IP addresses are PII: apply 2-year retention + restricted access controls

Scraping Ethics

  • Scrape publicly listed business information only (name, address, phone, service categories)
  • Respect robots.txt on all scraped domains; document compliance in scraper/ETHICS.md
  • Rate limit: ≥ 2s delay between requests; randomise user-agent string
  • Provide takedown mechanism: email address for removal requests from listed businesses

Transport & Storage Encryption

  • All API calls HTTPS (TLS 1.2+); Supabase enforces by default
  • Supabase Storage and PostgreSQL: AES-256 encryption at rest; enabled by default on managed instances
  • Vercel serves all assets over HTTPS; configure HSTS header

23. Cost & Scaling Notes

Expected Cost Drivers & Mitigations

DriverMitigation
FastAPI ML service (Railway)Railway free tier covers low-traffic academic use (~500 req/day). At scale, upgrade to Railway Starter ($5/mo). For very high volume, migrate ML inference to Supabase Edge Function with ONNX.js to eliminate the separate service.
Supabase row storage (assessments)Assessments are small rows (~1 KB). 100k assessments ≈ 100 MB. Archive assessments older than 1 year to cold storage. Partition audit_logs by month.
Supabase Storage egress (guide PDFs)Guide PDFs are static. Mirror to Vercel CDN edge cache with 24-hour TTL. CDN caching eliminates most repeated Storage egress charges.
Google Places API costsNearby Search: $0.032/request. places_cache (24h TTL) eliminates repeat calls for same area. At scale, batch common Metro Manila search areas into a nightly cache-warmer job to pre-populate cache before peak hours.
OSM tile bandwidthNo service worker tile caching (app is browser-only). At scale, consider self-hosting Protomaps static tiles on Cloudflare R2 (one-time map extract, near-zero egress cost) or using a tile CDN.
Vercel serverless invocations (scoring + directory)One scoring invocation per assessment; directory invocations depend on search frequency. At scale (>100k/month), migrate scoring logic to a Supabase Edge Function (closer to DB, lower cold-start latency).
Nominatim geocodingUsed only as fallback and for background geocoding of shop records. Cache geocoded addresses in DB; batch-geocode at registration time. At high volume, use OpenCage or Pelias instead (usage policy: 1 req/s max).
Supabase connections (serverless)Enable PgBouncer transaction-mode pooling to prevent connection exhaustion from concurrent Vercel function invocations.

Scalability Checklist

  • Directory search: paginate at 20–50 results/page; never return full table
  • GIST index on geom column: essential for sub-100ms queries at 10k+ records
  • Scoring config cached in serverless function memory: avoid DB round-trip on every assessment
  • Vite fingerprints all JS/CSS assets: ensures browser caches are invalidated on deploy
  • Google Places cache hit rate should be monitored; low hit rate indicates cache key is too granular — consider rounding lat/lon to 2 decimal places (~1 km grid)
  • Enable read replicas when directory query volume exceeds Pro plan connection limits
Recommended first milestone: Ship Assess (rule-based Phase 1 scoring) + Navigate with a static, manually seeded Connect directory. This delivers core user value immediately, begins collecting user_confirmed_outcome data, and lets the team validate the scoring baseline before training any ML model. ML is introduced in Phase 1.5 (clustering) once the pipeline is stable.
